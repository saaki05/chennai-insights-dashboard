# Real-Time Chennai Insights Dashboard

A smart-city traffic dashboard for Chennai: a live (simulated) GPS/traffic
feed across 28 real arterial road segments, DBSCAN density clustering to
flag congestion hotspots in real time, short-horizon forecasting, incident
simulation (accidents/waterlogging/roadwork/events), and an interactive
Leaflet/OpenStreetMap map — no paid API key required anywhere in the stack.

## Why simulated data, not a live feed?

There is no free, no-signup, real-time GPS/traffic API for Chennai
specifically. Rather than gate the whole project behind a billed Google
Maps/TomTom account, `backend/simulator.py` generates realistic synthetic
traffic on the *real* Chennai road network (actual junction coordinates,
actual road names) using a time-of-day / day-of-week congestion model
(morning + evening rush, IT-corridor shift-traffic peaks, monsoon "rain
mode", random incidents). The output schema matches what a real provider
would return, so swapping in a live feed later only means replacing
`TrafficSimulator.tick()` — nothing downstream (clustering, API, frontend)
needs to change.

## Architecture

```
config/chennai_network.py   Real Chennai zones + road segments (ground truth)
backend/simulator.py        Synthetic real-time GPS/congestion generator
backend/clustering.py       DBSCAN hotspot detection (haversine, congestion-weighted)
backend/forecasting.py      Rolling linear-trend short-horizon forecast per road
backend/database.py         SQLite persistence (aggregated snapshots + hotspot events)
backend/ws_manager.py       WebSocket broadcast to connected dashboard clients
backend/main.py             FastAPI app: REST API + /ws/live + background sim loop
frontend/                   Leaflet (OpenStreetMap) + Chart.js dashboard, vanilla JS
notebooks/research_analysis.ipynb   Offline clustering/forecasting research notebook

scripts/generate_static_site.py   Static-snapshot builder for the GitHub Pages deploy
docs/                              Static frontend (polls JSON, no WebSocket) + docs/data/*.json
.github/workflows/update-snapshot.yml   Cron: regenerates docs/data/*.json every 15 min
```

## Live deployment: GitHub Pages (free, zero setup)

**https://saaki05.github.io/chennai-insights-dashboard/**

GitHub Pages only serves static files — there's no server to run the live
WebSocket loop on. So `docs/` is a separate static build of the dashboard:
a [GitHub Actions workflow](.github/workflows/update-snapshot.yml) runs
every 15 minutes on GitHub's free schedule, executing
[`scripts/generate_static_site.py`](scripts/generate_static_site.py) — which
reuses the *exact same* `backend.simulator` / `backend.clustering` /
`backend.forecasting` modules as the live app — and commits the resulting
JSON (`docs/data/*.json`) back to the repo. The static frontend polls that
JSON every 20s instead of a WebSocket. Net effect: near-real-time (15-min
cadence) instead of second-by-second, but $0 cost, no card, no account
beyond GitHub, and no server to maintain.

To point it at your own fork: enable Pages in repo Settings → Pages →
deploy from `main` / `/docs`, and enable Actions if they're off by default.

## Running the full real-time version locally

If you want true second-by-second updates (WebSocket push, background
simulation loop, SQLite history) rather than the 15-min static snapshot,
run the original FastAPI app — it needs an actual long-running process, so
serverless hosts (Vercel/Netlify) and static hosts (GitHub Pages) can't run
it; deploy it to any host that keeps a Python process alive continuously
(a VM/VPS or a "web service"-style platform) if you want it always-on.

```bash
pip install -r requirements.txt
uvicorn backend.main:app --reload --port 8000
```

Then open http://localhost:8000 — the map, hotspot list, alerts feed, and
per-road forecast chart update live over a WebSocket every ~3 seconds. API
docs (auto-generated by FastAPI) are at http://localhost:8000/docs.

## Key endpoints

| Endpoint | Purpose |
|---|---|
| `GET /api/meta` | Zones, roads, map bounds/center |
| `GET /api/snapshot` | Current pings + hotspots + incidents |
| `WS /ws/live` | Streaming snapshot every simulator tick |
| `GET /api/history/{road_id}?minutes=30` | Historical congestion for a road |
| `GET /api/forecast/{road_id}` | 5/15/30-min congestion forecast |
| `GET /api/hotspot-events?minutes=60` | Recent clustered hotspot history |
| `POST /api/research/eps-sweep` | Run DBSCAN over a range of `eps` on the live snapshot |
| `POST /api/rain/toggle` | Simulate a monsoon rain event (raises congestion city-wide) |

## Research friendliness

- `notebooks/research_analysis.ipynb` reuses the exact same simulator/clustering
  modules as the live app (no duplicated logic) to run k-distance elbow plots,
  an eps/min_samples sensitivity sweep, a DBSCAN-vs-KMeans comparison, and a
  full simulated 24h congestion-cycle analysis per road category.
- `sweep_eps()` in `backend/clustering.py` and the `/api/research/eps-sweep`
  endpoint let you interrogate the live feed's clustering sensitivity without
  leaving the running dashboard.
- The synthetic model is fully deterministic given a seed
  (`TrafficSimulator(rng_seed=...)`), so experiments are reproducible.
- Swapping in real data later (TomTom/HERE/Google Roads) only requires
  matching the ping schema — everything else (clustering, forecasting, API,
  frontend) is provider-agnostic.

## Tech stack

FastAPI + WebSockets, scikit-learn (DBSCAN), NumPy/Pandas, SQLite
(`aiosqlite`), vanilla JS + Leaflet.js/Leaflet.heat (OpenStreetMap tiles,
free, no key) + Chart.js. No Google Maps dependency, no billing account
required anywhere.
